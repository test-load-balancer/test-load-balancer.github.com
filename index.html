What is Test Load Balancer (TLB)?

Test Load Balancer is a tool that can automatically partition tests among multiple sets, each one of which can run on different physical machine. This can significantly improve the speed of a test stage in a pipeline, and significantly shorten the feedback loops for large or slow running test suites such as functional or UI tests.

In addition to balancing, it also allows you to do other interesting things like re-order your tests-subset, before it executes. For instance, run tests that failed in the last run, first.

How does TLB work?

TLB has two primary components.

  • A Server: that can stores and allows querying of test data(test times/test results etc)
  • A Balancer: that knows how to partition and re-order suite of tests, given a server url
Balancer hooks-up with your build-framework and testing-framework to do the actual work, whereas Server is primarily a data-repository that balancer talks to. Data from previous run(s) is used by balancer to partition/re-order the current run, and data from the current run is posted back so it can be used for future runs.

Can TLB be used for test suite written in a non-java/non-jvm language(Ruby, C#, Python etc..)?

Yes. While TLB is written in Java, it was written from ground up to support both jvm and non-jvm based languages/runtimes. Balancer is capable of running as a standalone process, in alien-environment(as we call it) setup framework/language that tests are written in does not matter. {% include tlb_rb_link %} for instance, runs on MRI(CRuby) and uses standalone balancer. However, since TLB-core is written in java, you will need to have java installed on the box running tests(so balancer process can be launched).

Infact, it feels like running balancer standalone is actually a better option in most cases, because it has the inherent advantage of process level isolation.
